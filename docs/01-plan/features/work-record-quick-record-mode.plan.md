## 1. Overview

### 1.1 Purpose

빠른 기록 방식 설정은 사용자가 출근/퇴근을 기록할 때 실제 근무 시각과 기록 버튼을 누른 시각이 달라지는 문제를 줄이기 위한 설정 후보이다.

현재 홈 화면과 상시 알림 액션은 버튼을 누른 현재 시각을 바로 저장한다. 이 방식은 10초 이내 기록에는 적합하지만, 정시 근무자, 유연근무자, 1분 단위 수당 지급 사용자처럼 실제 근무 시작/종료 시각을 정확히 남기고 싶은 사용자에게 반복적인 수동 수정 부담을 만든다.

이 Plan은 자동 보정 기능을 확정하지 않는다. 사용자가 명시적으로 선택하는 빠른 기록 방식이 MVP에 적합한지 확인하고, 다음 Design 단계에서 데이터 모델과 UX를 검토할 범위를 정한다.

### 1.2 Background

WorkLedger MVP의 핵심 가치는 10초 이내 근무 기록, 상시 알림 액션, 로컬 저장, 수동 수정이다. 기존 MVP는 회사 시스템 연동, 법률/증거 보장, 자동 수당 계산을 제외하고 개인 기록의 지속성을 검증하는 방향으로 완료되었다.

이번 개선 후보는 기존 MVP의 방향을 바꾸지 않고, 빠른 기록 중 발생하는 시각 선택 마찰을 줄이는 작은 설정이다. 특히 “근로제도 판정”이 아니라 “기록 방식 선택”으로 표현해야 한다.

## 2. User Intent

| 항목 | 내용 |
|---|---|
| 핵심 문제 | 출근/퇴근 버튼을 누른 현재 시각만 저장하면 실제 근무 시각과 기록 시각이 어긋나 반복 수동 수정이 필요하다 |
| 대상 사용자 | 정시 출퇴근 사용자, 유연근무 사용자, 1분 단위 기록을 중요하게 보는 사용자 |
| 핵심 니즈 | 빠른 기록의 속도는 유지하면서 사용자가 원하는 기록 시각을 명시적으로 선택하고 싶다 |
| 성공 모습 | 기록 1회가 10초 이내에 끝나고, 기록 후 수동 수정 빈도가 줄어든다 |
| 제약 | 로컬 저장 유지, 계정/서버/GPS/회사 시스템 연동 제외, 자동 법률 판단 및 자동 수당 계산 제외 |

## 3. Alternatives Explored

| 접근안 | 설명 | 장점 | 단점 | 노력 | 판단 |
|---|---|---|---|---|---|
| A. 현재 시각 즉시 저장 유지 | 홈/알림 액션이 지금 시각을 바로 저장하고 필요 시 사용자가 기존 수정 화면에서 고친다 | 구현 변경 없음, 10초 기록 목표에 가장 단순, 오작동 여지 낮음 | 실제 시각과 기록 시각이 자주 다르면 수동 수정 부담이 지속됨, 정시 사용자에게 반복 마찰 발생 | Low | 기준선으로 유지하되 개선 필요 |
| B. 설정 기반 후보 시각 표시 | 설정에서 빠른 기록 방식을 고르고, 기록 시 현재 시각 또는 정시 후보를 사용자가 명시 선택한다 | 자동 보정 오해를 줄임, 정시 사용자 수정 부담 감소, 기존 WorkRule 활용 가능 | 선택 UI가 늘어나 10초 목표를 해칠 수 있음, 알림 액션 UX 설계가 필요함 | Medium | Design 단계 우선 검토 |
| C. 1분 단위 직접 입력 우선 | 빠른 기록 후 즉시 1분 단위 시간 입력을 유도한다 | 정확 기록 욕구에 맞음, 유연근무/수당 사용자가 이해하기 쉬움 | 빠른 기록이라기보다 수정 플로우가 되어 마찰 증가, 상시 알림 장점 약화 | Medium | 보조 흐름으로 검토 |
| D. 기록 시각과 실제 시각 분리 | 버튼 누른 시각과 사용자가 인정한 실제 근무 시각을 별도 저장한다 | 감사/분석 관점에서 명확함, 추후 신뢰도 설명 가능 | 데이터 모델과 UI 복잡도 증가, MVP 범위를 넘을 수 있음 | High | 이번 MVP 개선에서는 보류 |

최종 Plan 기준 접근은 B를 중심으로 하되, A를 기본 기준선으로 유지하고 C의 직접 수정 접근을 보조 흐름으로 남기는 것이다. D는 후속 검증 전까지 제외한다.

## 10. Architecture Considerations

현재 코드 기준 검토 지점은 다음과 같다.

| 영역 | 현재 상태 | Design 단계 질문 |
|---|---|---|
| `WorkRecordRepository.clockIn()` | 저장소 clock의 현재 시각으로 출근 저장 | 선택된 후보 시각을 받는 별도 입력 모델이 필요한가 |
| `WorkRecordRepository.clockOut()` | 저장소 clock의 현재 시각으로 퇴근 저장 | 퇴근 후보 시각이 출근보다 빠른 경우 오류를 어떻게 보여줄 것인가 |
| 홈 기록 버튼 | 현재 primary action에 따라 즉시 `clockIn()` 또는 `clockOut()` 호출 | 설정 방식에 따라 후보 선택 화면이 필요한가 |
| 상시 알림 액션 | 알림 액션에서 바로 repository action 실행 | 알림에서는 즉시 저장을 유지할지, 앱을 열어 선택하게 할지 결정 필요 |
| `WorkRule` | 정시 출근/퇴근, 연장/야간 기준 저장 | 정시 후보 표시가 이 값을 읽어도 되는가 |
| 설정 화면 | 근무 설정, 총 연차, 알림 설정으로 구성 | 빠른 기록 방식 설정을 어느 설정 그룹에 둘 것인가 |

Design 단계에서는 새 모델이 필요하다면 이름을 “근로제도”가 아니라 `QuickRecordMode` 또는 이에 준하는 “기록 방식” 의미로 정의해야 한다.

## 11. Next Design Questions

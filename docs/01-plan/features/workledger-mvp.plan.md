# WorkLedger MVP Plan

| 항목 | 내용 |
|---|---|
| 한국어 앱 이름 | 내근무장부 |
| 영어 앱 이름 | WorkLedger |
| 기준 플랫폼 | Flutter Android |

## 1. User Intent

이 앱은 포괄임금제/고정OT 또는 야근이 잦은 사무직, IT, 스타트업 직장인을 위한 개인 근무 방어 기록 앱이다.

사용자는 야근, 초과근무, 휴일근무, 퇴근 지연, 연차 사용을 회사 시스템이나 기억에만 의존하지 않고 개인적으로 빠르게 남기고 싶어한다. 핵심 사용 맥락은 월말 또는 분쟁/정산 상황에서 내가 실제로 얼마나 일했고, 연차를 얼마나 썼는지 확인할 수 있어야 한다는 것이다.

MVP의 목적은 완전한 급여 계산기나 법률 증거 앱을 만드는 것이 아니라, 사용자가 10초 이내에 근무 기록을 남기고 월간 요약과 리포트 생성 의향을 통해 실제 기록 행동과 지불 의향을 검증하는 것이다.

| 항목 | 내용 |
|---|---|
| 핵심 문제 | 회사 시스템/기억에만 의존하면 실제 근무시간, 야근, 휴일근무, 잔여연차를 나중에 확인하기 어렵다 |
| 핵심 사용자 | 포괄임금제/고정OT 또는 야근이 잦은 사무직, IT, 스타트업 직장인 |
| 핵심 행동 | 출근, 퇴근, 야근, 휴일근무, 연차 사용을 10초 이내 기록 |
| 핵심 가치 | 월말에 개인 근무 기록, 초과근무, 잔여연차를 확인하고 리포트 필요성을 검증 |
| 제외 방향 | 법률 자문, 증거 효력 보장, 회사 시스템 연동, 복잡한 자동 수당 계산 |

## 2. Alternatives Explored

MVP 접근안은 3가지를 비교했다.

| 접근안 | 설명 | 장점 | 단점 | 판단 |
|---|---|---|---|---|
| A. 10초 기록 중심 MVP | 출근/퇴근/야근/휴일근무/연차 사용을 빠르게 기록하고 월간 요약을 제공 | 2주 MVP에 적합, 핵심 행동 검증 가능, 로컬 저장으로 가능 | 단순해 보일 수 있고, 결제 가치가 약할 수 있음 | 기본 축으로 채택 |
| B. 월말 리포트/방어 문서 중심 MVP | 월간 PDF/CSV 리포트와 결제 의향 검증을 중심에 둠 | 유료 전환 포인트가 명확하고, 방어형 기록이라는 차별화가 강함 | 기록 습관이 없으면 리포트 가치가 약함 | fake-door로 일부 채택 |
| C. 근무시간+잔여연차 통합 관리 앱 | 야근, 휴일근무, 잔여연차를 함께 관리하는 개인 근무 대장 | 일반 직장인에게 이해가 쉽고 사용 이유가 넓어짐 | 범위가 넓어지고 자동 연차 계산 요구가 생길 수 있음 | 수동 연차 기록만 채택 |

최종 선택은 A+B+C 조합이다.

> 10초 기록 중심 MVP를 기본으로 만들고, 월말 리포트는 fake-door로 결제/니즈 신호를 측정하며, 잔여연차는 자동 계산이 아닌 수동 기록으로 포함한다.

이 선택은 2주 MVP, 1인 개발, Flutter, 로컬 저장, 월 2만 원 이하 예산이라는 제약 안에서 핵심 리스크를 가장 빠르게 검증하기 위한 결정이다. 핵심 리스크는 기능 완성도가 아니라 직장인이 실제로 7일 중 4일 이상 기록하는지와 월말 리포트에 돈을 낼 신호가 있는지다.

| 교대근무표 자동 계산 | Won't Do | 초기 타깃 밖, 복잡도 큼 |

최종 MVP 핵심 플로우는 다음이다.

1. 앱 실행 또는 상시 알림 액션 사용
2. 오늘 출근/퇴근 1탭 기록
3. 필요 시 야근, 휴일근무, 퇴근 지연 태그 선택
4. 메모 1줄 입력
5. 연차 사용 시 사용 일수 기록
6. 월간 화면에서 근무시간, 초과근무, 잔여연차 확인
7. 월간 리포트 만들기 클릭
8. 가격표/fake-door에서 결제 의향 측정

## 4. Scope Definition

MVP는 10초 이내 개인 근무 기록을 가장 중요한 제품 경험으로 둔다. 사용자는 앱을 열거나, 상시 알림의 출근 기록/퇴근 기록 액션을 눌러 빠르게 기록할 수 있어야 한다.

| 범위 | 포함 |
|---|---|
| 핵심 기록 | 출근, 퇴근, 퇴근 지연, 야근, 휴일근무 |
| 연차 관리 | 총 연차 수동 입력, 연차 사용 기록, 잔여연차 표시 |
| 기록 보정 | 수동 시간 수정, 메모 입력 |
| 요약 | 월간 근무시간, 월간 초과근무, 연차 사용/잔여 요약 |
| 빠른 입력 | 상시 알림 + 출근 기록, 퇴근 기록 알림 액션 |
| 수익 검증 | 월간 리포트 버튼, 가격표 화면, Report Pass/Pro 클릭 측정 |
| 저장 방식 | 계정 없이 로컬 저장 |
| 언어 | 한국어 우선, 영어 i18n 구조만 준비 |

| 제외 | 이유 |
|---|---|
| 로그인/클라우드 동기화 | 비용과 보안 복잡도 증가 |
| GPS 자동 기록 | 개인정보, 배터리, 정확성 리스크 |
| 회사 근태 시스템 연동 | 초기 MVP 범위 초과 |
| 법률 자문/증거 효력 보장 | 책임 리스크 |
| 자동 법정 연차 계산 | 회사 정책 차이로 오류 위험 |
| 복잡한 수당 자동 계산 | 정확성 기대와 법률 리스크 증가 |
| 교대근무표 자동 계산 | 초기 타깃 밖 |
| Quick Settings Tile / 홈 위젯 | 빠른 입력에는 좋지만 2주 MVP 후순위 |

핵심 판단은 다음이다.

> MVP는 정확한 급여 계산 앱이 아니라, 개인이 빠르게 근무, 야근, 연차 기록을 남기고 월말에 확인하는 앱이다.

## 6. Success Criteria

MVP 성공 여부는 기능 완성도가 아니라 실제 기록 행동, 리포트 니즈, 결제 신호로 판단한다.

| 구분 | 성공 기준 | 측정 방식 |
|---|---:|---|
| 7일 기록 지속성 | 테스트 사용자 중 40% 이상이 7일 중 4일 이상 기록 | 로컬 기록 로그 또는 테스트 설문 |
| 핵심 행동 속도 | 기록 1회 완료 시간이 10초 이내 | 사용성 테스트, 화면 녹화, 자체 타이머 |
| 리포트 니즈 | 사용자 50% 이상이 월간 PDF/CSV 리포트가 필요하다고 응답 | 테스트 종료 설문 |
| 결제 신호 | 리포트 생성/Pro 버튼 클릭률 10% 이상 | 앱 내 클릭 이벤트 |
| 유료 의향 | 가격표 도달자 중 3~5% 이상이 결제 의향 클릭 | fake-door 가격표 클릭 |
| 리텐션 | D7 재방문율 25% 이상 | 로컬 실행 로그 또는 테스트 설문 |
| 주관적 가치 | 회사 시스템 외에 개인 기록이 필요하다는 응답 60% 이상 | 테스트 종료 설문 |

### Go / No-Go

| 판단 | 기준 |
|---|---|
| Go | 7일 중 4일 이상 기록률 40% 이상, 리포트 필요 응답 50% 이상, 리포트/Pro 클릭률 10% 이상 중 2개 이상 충족 |
| Iterate | 기록 지속성은 낮지만 리포트 니즈 또는 결제 신호가 강함 |
| Pivot | 기록 지속성, 리포트 니즈, 결제 신호가 모두 기준 미달 |
| Drop | 사용자가 개인 기록 필요성 자체를 낮게 평가하고, 7일 테스트 참여도도 낮음 |

## 7. Risks And Mitigations

| 리스크 | 영향 | 완화책 |
|---|---|---|
| 사용자가 매일 기록하지 않음 | 핵심 가치가 쌓이지 않음 | 상시 알림 액션으로 기록 마찰을 줄이고, 7일 테스트에서 기록 지속성 먼저 검증 |
| 앱이 단순해 보여 설치 가치가 약함 | 초기 전환율 저하 | 10초 근무 방어 기록, 월말에 내 야근 확인으로 메시지 집중 |
| 리포트 결제 의향이 낮음 | 수익성 약화 | 실제 결제 전 fake-door 가격표로 Report Pass/Pro 클릭률 검증 |
| 수당 계산 정확성 기대가 커짐 | 불만/법률 리스크 | 참고용 개인 기록으로 표현하고 자동 임금 산정은 MVP에서 제외 |
| 연차 계산 오류 | 신뢰 하락 | 법정 자동 계산 대신 총 연차/사용 연차 수동 입력만 제공 |
| 상시 알림이 사용자에게 부담스러움 | 알림 거부/삭제 | 알림은 명확한 온보딩 후 선택형으로 제공하고, 앱 내부 기록은 알림 없이도 가능하게 유지 |
| Android 알림 정책/권한 문제 | 빠른 기록 기능 실패 | 알림 권한 거부 시 대체 경로 제공, 앱 홈 1탭 기록 유지 |
| 개인정보/민감정보 우려 | 설치/사용 회피 | 계정 없이 로컬 저장, 회사명/위치 필수 입력 없음 |
| 교대근무/알바 사용자의 계산 요구 | 범위 확장 압박 | 초기 타깃 밖으로 명시하고 Pro/후속 확장 후보로 분리 |
| 경쟁 앱과 차별화 부족 | 시장 진입 약화 | 한국 직장인 야근/연차 방어 기록, 월말 리포트, 상시 알림 10초 입력으로 차별화 |

가장 큰 리스크는 직장인이 실제로 7일 중 4일 이상 기록하지 않을 가능성이다. 이 리스크가 해소되지 않으면 PDF/CSV, Pro, 수당 계산, 백업 같은 후속 기능은 의미가 약해진다. 따라서 MVP의 1차 검증은 기능 만족도가 아니라 기록 지속성이다.

## 8. Architecture Considerations

MVP는 Flutter Android 우선, 로컬 저장, 계정 없음, 서버 없음을 기본 구조로 한다.

| 영역 | 결정 |
|---|---|
| 앱 프레임워크 | Flutter |
| 우선 플랫폼 | Android |
| 상태 관리 | 단순 로컬 상태 관리. MVP에서는 과도한 아키텍처 패턴을 피함 |
| 데이터 저장 | 로컬 DB 또는 로컬 key-value 저장 |
| 계정 | 없음 |
| 서버 | 없음 |
| 알림 | Android 로컬 알림 + 상시 알림 액션 |
| 분석 | 서버 분석 도구 없이 로컬 이벤트 로그 또는 테스트용 수동 export |
| i18n | 한국어 기본, 영어 문자열 구조만 준비 |
| 결제 | 실제 인앱결제 없음. 가격표/fake-door 클릭 측정만 구현 |
| 리포트 | MVP에서는 실제 PDF/CSV 생성보다 리포트 버튼과 가격표 전환 우선 |

### Data Model Draft

| Entity | 주요 필드 |
|---|---|
| WorkRecord | 날짜, 출근 시간, 퇴근 시간, 태그, 메모, 생성/수정 시간 |
| LeaveBalance | 총 연차, 사용 연차, 잔여 연차, 기준 연도 |
| LeaveUsage | 사용 날짜, 사용 일수, 메모 |
| PricingIntentEvent | 이벤트 유형, 선택 요금제, 발생 시간 |

### Notification Design

| 항목 | 내용 |
|---|---|
| 상시 알림 | 사용자가 켜면 알림창에 유지 |
| 알림 액션 | 출근 기록, 퇴근 기록 |
| 권한 거부 대응 | 앱 홈의 1탭 기록은 계속 동작 |
| 잠금화면 표현 | OS 정책에 따라 제한될 수 있으므로 빠른 기록으로 표현 |
| 후순위 | Quick Settings Tile, 홈 위젯 |

### Implementation Principles

- 입력 흐름은 10초 이내 완료를 기준으로 설계한다.
- 기록 데이터는 사용자의 명시 입력 없이 외부 전송하지 않는다.
- 법률/임금 계산 정확성을 보장하는 로직과 문구를 넣지 않는다.
- 연차는 자동 계산하지 않고 사용자가 입력한 값만 기준으로 표시한다.
- 복잡한 수당 계산, 교대근무표, 회사 시스템 연동은 후속 확장으로 분리한다.
